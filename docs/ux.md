# QQ Group Bot — UX Templates

All templates use `{variable}` placeholders for string interpolation.
Tone: friendly 群友 for chat; strict but respectful for moderation.

---

## 1. /help — Full Text

```
欢迎使用群机器人！以下是所有可用指令：

【聊天 & 模仿】
/mimic @群友 [话题]   — 让我模仿某位群友的说话风格回复一句
/mimic_on @群友       — 开启持续模仿模式（全群生效，任何人可关闭）
/mimic_off            — 关闭当前的模仿模式

【群规 & 管理】（仅管理员）
/rule_add <描述>       — 添加一条群规或违规示例
/rule_false_positive <消息ID> — 标记一条 AI 误判，撤销对应处罚

【申诉】
/appeal               — 申诉你最近一次受到的处罚（处罚后24小时内有效）

【查看信息】
/rules                — 查看本群当前所有群规
/stats                — 查看本群近7天的统计数据

如有疑问请联系群管理员。
```

## 2. 各指令回复模板

### /mimic @user [topic]

**成功：**
```
{generated_reply}
```

**错误 — 无历史记录（E002）：**
```
@{nickname} 在本群没有历史消息记录，无法进行模仿。
```

**警告 — 历史太少（E003，仍继续执行）：**
```
⚠️ @{nickname} 目前只有 {count} 条历史消息，模仿效果可能不准确。

{generated_reply}
```

**错误 — 模仿自己（E017）：**
```
我没办法模仿我自己啦。
```

**成功：**

## 4. 申诉流程 UX

### 用户发起申诉

用户在群内发送 `/appeal`。

**Bot 回复（成功）：**
```
@{user} 申诉已提交，禁言已暂时解除，等待管理员审核。
```

**同时私信通知管理员（或群内通知）：**
```
[申诉通知] @{user} 对以下处罚提出申诉：
- 处罚类型：{punishment_type}
- 处罚时间：{punishment_time}
- AI 判定原因：{ai_reason}

请使用 /rule_false_positive {msg_id} 批准申诉，或（将来）使用 /appeal_deny {msg_id} 拒绝。
```

```
@{user} 申诉成功，处罚已撤销。感谢你的反馈，我们已将此情况记入学习库。
```

### 管理员拒绝（/appeal_deny，第二阶段功能）

```
@{user} 申诉未获批准，原处罚维持有效。如有疑问请联系管理员。
```

## 9. /char — 角色扮演指令 UX

### 9.1 设计决策：激活回复格式

**问题**：`/char` 成功激活时，bot 应回复"切换通知"还是直接用角色声音说第一句话？

**选项 A：元通知（推荐）**

```
已切换至角色模式：凑友希那 (ykn)。使用 /char_off 关闭。
```

优点：
- 管理员明确知道命令已生效，不会怀疑是否成功
- 在群里不突兀，因为这条消息明显是"系统通知"，不是角色在说话
- 错误排查更容易（能区分"激活失败"和"角色沉默"）

缺点：
- 破坏沉浸感，第一句话是机器人口吻而非角色

**选项 B：角色首句（沉浸式）**

激活后直接以角色口吻说一句台词，例如凑友希那会说：
```
……总算叫到我了。
```

优点：
- 沉浸感强，角色切换即生效，氛围到位
- 对懂 BanG Dream 的群友更有趣

缺点：
- 管理员不能确认命令是否生效（角色可能 `<skip>` 了首句）
- 角色首句由 LLM 生成，有概率不在角色状态（刚加载 lore，首句风险略高）
- 如果是切换角色，群友可能看不懂谁在说话

**结论**：推荐选项 A（元通知）作为默认行为。沉浸感通过后续自然对话建立，不需要激活时强行"入戏"。若将来想要沉浸式开场，可作为可选配置项（`/char set ykn --immersive`）。

---

### 9.2 回复前缀决策：是否标记角色名？

**问题**：`/mimic` 生成的回复不加前缀（见 section 5）。`/char` 回复是否加 `[{character_name}]` 前缀？

**推荐：不加前缀，完全隐式（与 mimic 一致）**

理由：
- 前缀破坏角色扮演的沉浸感——凑友希那不会在说话前自报名字
- 群友自然会从文风和内容判断当前是哪个角色
- `/char status` 随时可以查看当前角色，不需要每条消息都标注
- 与现有 `/mimic` 的"无前缀"规则保持一致，代码路径更简单

前缀 `[凑友希那]` 仅保留在服务端 pino 日志中（info 级别），不出现在群消息里。

### 故障安全静默策略

sticker-first 模式内部所有失败路径（表情包库为空、embedder 未就绪、所有候选分数低于阈值、所有候选在冷却期、文件不存在、DB 读取异常、未捕获异常）均**静默回退为文字回复**，不向群内发送任何额外提示消息。错误记录到 bot 日志（WARN 或 ERROR 级别），用户感知不到模式内部的降级行为。
