# QQ Group Bot — UX Templates

All templates use `{variable}` placeholders for string interpolation.
Tone: friendly 群友 for chat; strict but respectful for moderation.

---

## 1. /help — Full Text

```
欢迎使用群机器人！以下是所有可用指令：

【聊天 & 模仿】
/mimic @群友 [话题]   — 让我模仿某位群友的说话风格回复一句
/mimic_on @群友       — 开启持续模仿模式（全群生效，任何人可关闭）
/mimic_off            — 关闭当前的模仿模式

【群规 & 管理】（仅管理员）
/rule_add <描述>       — 添加一条群规或违规示例
/rule_false_positive <消息ID> — 标记一条 AI 误判，撤销对应处罚

【申诉】
/appeal               — 申诉你最近一次受到的处罚（处罚后24小时内有效）

【查看信息】
/rules                — 查看本群当前所有群规
/stats                — 查看本群近7天的统计数据

如有疑问请联系群管理员。
```

## 2. 各指令回复模板

### /mimic @user [topic]

**成功：**
```
{generated_reply}
```

**错误 — 无历史记录（E002）：**
```
@{nickname} 在本群没有历史消息记录，无法进行模仿。
```

**警告 — 历史太少（E003，仍继续执行）：**
```
⚠️ @{nickname} 目前只有 {count} 条历史消息，模仿效果可能不准确。

{generated_reply}
```

**错误 — 模仿自己（E017）：**
```
我没办法模仿我自己啦。
```

**成功：**

## 4. 申诉流程 UX

### 用户发起申诉

用户在群内发送 `/appeal`。

**Bot 回复（成功）：**
```
@{user} 申诉已提交，禁言已暂时解除，等待管理员审核。
```

**同时私信通知管理员（或群内通知）：**
```
[申诉通知] @{user} 对以下处罚提出申诉：
- 处罚类型：{punishment_type}
- 处罚时间：{punishment_time}
- AI 判定原因：{ai_reason}

请使用 /rule_false_positive {msg_id} 批准申诉，或（将来）使用 /appeal_deny {msg_id} 拒绝。
```

```
@{user} 申诉成功，处罚已撤销。感谢你的反馈，我们已将此情况记入学习库。
```

### 管理员拒绝（/appeal_deny，第二阶段功能）

```
@{user} 申诉未获批准，原处罚维持有效。如有疑问请联系管理员。
```

## 9. /char — 角色扮演指令 UX

### 9.1 设计决策：激活回复格式

**问题**：`/char` 成功激活时，bot 应回复"切换通知"还是直接用角色声音说第一句话？

**选项 A：元通知（推荐）**

```
已切换至角色模式：凑友希那 (ykn)。使用 /char_off 关闭。
```

优点：
- 管理员明确知道命令已生效，不会怀疑是否成功
- 在群里不突兀，因为这条消息明显是"系统通知"，不是角色在说话
- 错误排查更容易（能区分"激活失败"和"角色沉默"）

缺点：
- 破坏沉浸感，第一句话是机器人口吻而非角色

**选项 B：角色首句（沉浸式）**

激活后直接以角色口吻说一句台词，例如凑友希那会说：
```
……总算叫到我了。
```

优点：
- 沉浸感强，角色切换即生效，氛围到位
- 对懂 BanG Dream 的群友更有趣

缺点：
- 管理员不能确认命令是否生效（角色可能 `<skip>` 了首句）
- 角色首句由 LLM 生成，有概率不在角色状态（刚加载 lore，首句风险略高）
- 如果是切换角色，群友可能看不懂谁在说话

**结论**：推荐选项 A（元通知）作为默认行为。沉浸感通过后续自然对话建立，不需要激活时强行"入戏"。若将来想要沉浸式开场，可作为可选配置项（`/char set ykn --immersive`）。

---

### 9.2 回复前缀决策：是否标记角色名？

**问题**：`/mimic` 生成的回复不加前缀（见 section 5）。`/char` 回复是否加 `[{character_name}]` 前缀？

**推荐：不加前缀，完全隐式（与 mimic 一致）**

理由：
- 前缀破坏角色扮演的沉浸感——凑友希那不会在说话前自报名字
- 群友自然会从文风和内容判断当前是哪个角色
- `/char status` 随时可以查看当前角色，不需要每条消息都标注
- 与现有 `/mimic` 的"无前缀"规则保持一致，代码路径更简单

前缀 `[凑友希那]` 仅保留在服务端 pino 日志中（info 级别），不出现在群消息里。

## 10. Bandori Live — 知识注入 UX 模板

> **功能定位**：无用户命令，始终开启。当触发消息含 Live 相关关键词时，bot 在生成回复前自动获取最多 3 条近期 BanG Dream! 演出信息，作为背景上下文注入，bot 自行决定是否自然提及——不强制。

---

### 10.1 注入块格式

注入块以固定标头开头，后跟最多 3 条演出条目，每条一行。格式简洁，专为 LLM 阅读设计，**不直接转发给用户**。

```
【近期 BanG Dream! Live 信息】
- {title}｜{startDate}（{endDate 存在时：~ endDate}）｜{bands}｜{venue}・{city}（{ticketInfoText 前40字，如有}）
```

**字段缺失时的降级规则**：

| 字段 | 缺失时显示 |
|------|-----------|
| `startDate` | `日程未定` |
| `venue` / `city` 均缺 | `场馆未定` |
| `bands` 为空 | `未知乐队` |
| `ticketInfoText` 缺失 | 省略括号部分 |

**完整示例**（供开发测试对照）：

```
【近期 BanG Dream! Live 信息】
- Roselia LIVE 2026 "Noir"｜2026-05-10 ~ 2026-05-11｜Roselia｜パシフィコ横浜・横浜（チケット一般発売中）
- MyGO!!!!! 夏巡 上海站｜2026-06-20｜MyGO!!!!!｜梅赛德斯奔驰文化中心・上海
- Ave Mujica 首次巡演｜日程未定｜Ave Mujica｜场馆未定
```

---

### 10.2 LLM 提示片段（用于 userContent 注入块前）

以下文字跟随注入块一起写入 `userContent`，置于注入块**之后、对话历史之前**，引导 LLM 自然使用上下文而非机械列举：

```
（以上是刚拿到的 Live 排期信息，仅供你参考。如果和话题相关，可以像群里的粉丝一样自然聊几句——比如说说哪场你期待或者票还有没有；如果话题无关，就当没看到。绝对不要把上面的信息当成列表原文输出，要融入对话语气。）
```

**核心约束**（开发者务必照字面写入，不得精简）：
- "像群里的粉丝一样自然聊几句" — 定义口吻基调
- "绝对不要把上面的信息当成列表原文输出" — 防止机械复读
- "如果话题无关，就当没看到" — 防止强行扯 Live 话题

### 10.3 用户主动询问排期时的回复基调

当用户明确问"有什么 live 排期""Roselia 最近有演出吗"等，bot 有了注入上下文后应**以粉丝口吻聊天，而非机械列举**。

**推荐语气示例**（非固定模板，LLM 可自由发挥同等风格）：

```
哦对，Roselia 五月好像有场"Noir"！横浜那边，两天连演，票好像还能抢来着？
```

```

```

```

```

**禁止输出的风格**（开发/测试对照，出现则视为 sentinel 失败）：

```
<!-- 错误示例 —— 机械列举，不得输出 -->

1. Roselia LIVE 2026 "Noir" | 2026-05-10 ~ 2026-05-11 | ...
2. MyGO!!!!! 夏巡 上海站 | ...

```

### 10.4 无数据时的静默策略

当 `getUpcoming()` 返回空数组（库为空、所有事件已过期、scraper 从未成功运行），注入块**不生成**，`liveBlock` 为空字符串，`userContent` 与无注入时完全相同。

**Bot 不应主动告知用户"暂时没有 Live 信息"**。无数据 = 静默，回复走正常邦批人格路径。如果用户追问"你知道有没有 live"而 bot 确实无数据，可以说：

```
这个我最近没看到消息哎，你去官推或者 bangdream.com 查下？
```

该回复由 LLM 自然生成，**不是硬编码模板**。注入块为空时 LLM 不应凭空捏造日期或场馆（诚实底线）。

---

### 10.5 新鲜度窗口与过期事件处理

#### 数据新鲜度

- Scraper 默认每 24 小时运行一次（`BANDORI_SCRAPE_INTERVAL_MS`），初次启动 60 秒后执行
- `getUpcoming(today, 3)` 过滤条件：`start_date >= today`（ISO 格式）
- `start_date IS NULL` 的事件**包含**在 `getUpcoming` 结果中（日程未定的活动仍值得告知）
- scraper 失败时保留上次成功数据，不清空；bot 可能以最多 24h 旧数据回复，属于可接受范围

#### 推荐注入窗口：未来 60 天内 + 进行中

**推荐**：`getUpcoming` 在 Architect 实现时应增加上限过滤 `start_date <= today + 60 days`，只注入 60 天内开始的演出；日期未知（NULL）事件无上限限制，始终包含。

**理由**：
- BanG Dream! live 通常在开演前 1–3 个月开票，60 天窗口覆盖绝大多数粉丝关心的"近期"演出
- 超过 60 天的活动信息对当前对话几乎无帮助，且官方票务细节尚未确定，注入后 LLM 可能产生误导性回复
- 进行中活动（`start_date <= today <= end_date`）已被 `start_date >= today` 覆盖（当天仍展示），无需额外逻辑

**cap 保持 3 条**：60 天窗口内超过 3 条时，仍只取最近 3 条（按 `start_date` 升序）。

#### 已过期事件的 bot 行为

`getUpcoming` 过滤掉 `start_date < today` 的事件，bot 在未来回复中不会主动提及已结束演出。

**若用户明确提到过去的演出**（"上次 Roselia 武道馆""去年那个 live"），关键词仍会触发注入，但注入块只包含近期事件。Architect 可选择额外查询最近 30 天内已结束的事件（`start_date >= today - 30 days AND start_date < today`）并附加"已完结"标记注入，格式如下：

```
【近期 BanG Dream! Live 信息】
- Roselia 10th☆LIVE "Woven"｜2026-03-15 ~ 2026-03-16｜Roselia｜さいたまスーパーアリーナ・さいたま【已完结】
- Roselia 11th☆LIVE｜2026-06-07｜Roselia｜日本武道館・東京
```

**v1 实现决策**：是否实现过去事件注入由 Architect 在 Iteration Contract 中决定。UX 要求如下：
- 若注入：格式如上，末尾追加`【已完结】`标签，LLM 看到后可自然聊"那场我也想去但没去成"等口吻
- 若不注入：bot 对历史话题只凭自身知识回应，不得捏造细节

```
哦那场！我没去 orz 听说最后返场唱了 black shout，羡慕死了
```

## 11. 表情包优先模式 (Sticker-First) — 完整 UX 模板

> **功能定位**：每群独立开关。开启后，bot 正常运行 LLM 生成意图文本，然后用该文本查找本地表情包库。找到高相似度表情包则以表情包代替文字发送；找不到则发文字。任何内部失败均静默回退为文字，用户感知不到。

### 11.4 /stickerfirst_status

**正常回复：**
```
【表情包优先模式状态】
开关: {ON | OFF}
匹配阈值: {threshold}
本群本地表情包库大小: {count} 张
最近发送表情包时间: {YYYY-MM-DD HH:mm | 暂无}
```

**字段细则：**

| 字段 | 显示规则 |
|------|---------|
| 开关 | `ON` 或 `OFF`，与 `sticker_first_enabled` 直接对应 |
| 匹配阈值 | 显示当前 `sticker_first_threshold` 数值，两位小数（如 `0.25`） |
| 本群本地表情包库大小 | 当前 `local_stickers` 表中本群的行数；为 0 时显示 `0 张` |
| 最近发送表情包时间 | bot 本地时区 `YYYY-MM-DD HH:mm` 格式；从未发送过时显示 `暂无` |

**非管理员：** 无回复，静默忽略。

---

### 11.5 设计决策：文字 vs 表情包的最终输出规则

**核心约束（开发者必须严格遵守）：**

1. 表情包优先模式只能**替换**文字回复为表情包，绝不能将非空回复变成空输出。
2. LLM 返回 `<skip>` 或空串时，`generateReply` 在拦截器触发前即返回 null — 表情包拦截器不会执行，也不会发送任何内容，这是正确行为。
3. 表情包和文字是**二选一**，不同时发送（combo 是 v1 非目标）。

**输出决策流（简化）：**

```
LLM 生成意图文本
  ↓ 为空 / <skip> → 直接返回 null（表情包拦截器不执行）
  ↓ 非空
sticker-first 未开启 → 发文字
sticker-first 开启 →
  embedder 未就绪 → 发文字
  库为空 → 发文字
  所有候选低于阈值 → 发文字
  所有候选在冷却期 → 发文字
  文件不存在 → 排除该条，继续评估剩余候选
  DB 异常 / 未捕获异常 → 静默 WARN/ERROR 日志，发文字
  找到有效表情包 → 发表情包（文字丢弃）
```

---

### 11.6 重复抑制行为（用户可感知部分）

- 同一张表情包在 5 分钟内不重复发送（per-group 冷却）。
- 冷却期内最优选被排除后，自动尝试次优选（若次优也超过阈值）。
- 所有候选均在冷却期时，静默回退发文字，不告知用户原因。
- 冷却状态仅在内存中，bot 重启后重置（5 分钟窗口可接受）。
- **用户不会收到任何"表情包冷却中"的提示**，体验上与文字回复无异。

---

### 11.7 与其他模式的交互（用户可感知行为）

| 并行模式 | sticker-first 行为 |
|---------|-------------------|
| `/mimic_on` 激活 | **不生效** — mimic 路径不经过 `generateReply`，拦截器无钩子点；bot 仍以文字模仿目标用户 |

### 13.2 统计摘要栏

数据来自 `GET /mod/stats`，页面加载时请求一次（不自动轮询）。审核操作后前端同步增减计数（不再请求 `/mod/stats`）。

```
总计  200    待审核  87 ●    已批准  91    已拒绝  22
```

- **待审核**数字旁显示橙色圆点 `●`，吸引视线。
- 所有数字仅展示，无点击交互。
- 数字为 0 时仍显示，不隐藏。
- `GET /mod/stats` 失败时显示 `—` 占位，不阻塞表格加载。

### 13.3 筛选栏

水平排列，`flex-wrap` 换行（手机端自动堆叠）。

| 控件 | 类型 | 说明 |
|------|------|------|
| 群组 | `<select>` | "全部群组" + 各群 group_id（从列表响应提取去重） |
| 严重度 | 复选框 ×5 | `☐ 1  ☐ 2  ☐ 3  ☐ 4  ☐ 5`，全不选 = 不过滤 |
| 状态 | 单选按钮 | `○ 全部  ○ 待审核  ○ 已批准  ○ 已拒绝` |
| 时间范围 | `<select>` | 最近 7 天 / 最近 30 天 / 全部（默认"全部"） |
| 应用 | `<button>` | 点击后重置到第 1 页，重新 fetch 列表 |

筛选状态保存在内存 `state.filters` 对象中（v1 不持久化到 URL）。

### 13.7 批准 / 拒绝按钮交互

```
用户点击 [批准] 或 [拒绝]
  ↓
按钮文字变 "…"，disabled（防重复提交）
  ↓
fetch POST /mod/:id/review  { "verdict": "approved" | "rejected" }
  ↓
成功 200:
  → 就地更新该行：审核状态徽章、行背景色
  → 前端同步更新统计栏计数
  → 按钮恢复可用（支持改变判断）
失败 4xx/5xx:
  → 按钮恢复可用
  → 行末尾显示红字"操作失败，请重试"，2 秒后自动消失
```

**按钮样式**：
- 批准：绿色边框 + 绿色文字（`#16a34a`）
- 拒绝：红色边框 + 红色文字（`#dc2626`）
- 背景透明，hover 时轻填充背景

### 13.8 行详情展开（Accordion）

点击行中任意单元格（除操作按钮外）展开详情，同时折叠其他已展开行。数据来自 `GET /mod/:id`（懒加载，首次点击时请求）。

**展开区内容**：

```
消息 ID: 12345     处罚时间: 2026-04-15 14:32:10

原始消息内容：
┌──────────────────────────────────────┐
│ 来，{link}，大家快来买！便宜！         │
└──────────────────────────────────────┘

图片：[缩略图 80×80px，点击新标签打开]（如有）

LLM 判断原因：
"用户发送了包含外部链接的重复广告内容，违反群规第2条。
 置信度: 0.92  严重度: 3"

申诉信息：（仅 appealed != 0 时显示）
申诉状态: 已提交  处理结果: 等待审核
```

**字段映射**：

| 展示项 | 数据字段 |
|--------|---------|
| 消息 ID | `msg_id` |
| 处罚时间 | `timestamp`（完整 `YYYY-MM-DD HH:mm:ss`） |
| 原始消息内容 | `message_content`（如存在） |
| 图片缩略图 | `image_url`（`<img>` max-width:80px；加载失败显示 `[图片加载失败]`） |
| LLM 原因 | `reason`（完整）+ `confidence` + `severity` |
| 申诉 | `appealed`（0/1/2）+ `reversed` |

展开详情加载失败时，展开区显示"加载详情失败"，不自动折叠。

### 13.13 JS 架构约束（面向开发者）

- **无框架，无构建步骤**：全部 JS 内联在 `<script>` 中，文件即为 `src/server/mod-review.html`。
- **状态对象**（单一可变对象，禁止散落全局变量）：
  ```js
  const state = {
    page: 1,
    limit: 20,
    filters: { group: '', status: 'all', severity: [], timeRange: 'all' },
    stats: { total: 0, unreviewed: 0, approved: 0, rejected: 0 },
    expandedRowId: null,
  };
  ```
- **事件委托**：`<tbody>` 上一个 `click` 监听器，通过 `data-id`、`data-action` 属性路由批准/拒绝/展开。
- **XSS 防护**：用户数据一律 `textContent` 赋值，禁止 `innerHTML` 拼接；`<img src>` 需通过白名单正则验证（`/^https?:\/\//`）。
- **统一 fetch 包装**：`async function apiFetch(path, opts)` 处理 JSON 解析 + 错误捕获 + 统一 content-type。

### 13.14 错误状态展示

| 场景 | UI 反应 |
|------|---------|
| `GET /mod/list` 失败 | 表格区域红色提示："加载失败，请刷新页面重试。" |
| `GET /mod/stats` 失败 | 统计栏显示 `—` 占位，不阻塞表格 |
| `POST /mod/:id/review` 失败 | 行末尾红字"操作失败，请重试"，2 秒消失 |
| `GET /mod/:id` 展开失败 | 展开区显示"加载详情失败"，不折叠行 |
| 网络断开 | 各请求独立走失败路径；无全局 offline 处理 |
